# Кузница историй

**Кузница историй** — веб-приложение для создания, структурирования, связывания и экспорта сценарных материалов для настольных RPG. Проект разрабатывается в рамках выпускной квалификационной работы по специальности **09.02.07 Информационные системы и программирование**.

Платформа ориентирована на ведущих настольных ролевых игр, авторов сценариев и небольшие игровые сообщества. Основная идея проекта — предоставить единое рабочее пространство, в котором можно готовить сценарии, кампании, карты, персонажей, предметы и печатные материалы без использования разрозненных документов, таблиц и графических файлов.

> Основной акцент проекта: **конструктор сценариев и материалов для ведущего**.  
> Социальные функции, друзья и чаты рассматриваются как дополнительный слой системы, а не как главный смысл приложения.

## Назначение проекта

Проект предназначен для цифровой подготовки материалов настольных RPG. Система помогает ведущему:

- создавать кампании и отдельные сценарии;
- собирать сценарий как graph-сценарий из связанных сцен-узлов и переходов;
- хранить персонажей, предметы, карты и связанные материалы;
- связывать между собой сцены, персонажей, предметы, карты и локации;
- экспортировать сценарий и игровые материалы в печатный формат;
- в перспективе публиковать отдельные материалы и обсуждать их в сообществе;
- в перспективе управлять приватностью и доступом к материалам.

Главная ценность приложения — не замена полноценного виртуального игрового стола, а удобная подготовка сценария и материалов до начала игровой сессии.

---

## Границы проекта

Проект развивается как **GM workspace first** — рабочее пространство ведущего для подготовки и сопровождения сценариев.

Это означает, что в центре системы находятся:

1. **Кампания** — верхний контейнер материалов игрового мира.
2. **Сценарий** — отдельное приключение, сессия или сюжетная ветка.
3. **Graph-сцены сценария** — узлы, переходы, исходы и preview-прохождение.
4. **Карты, персонажи, предметы и сущности мира** — связанные материалы, используемые в сценарии.
5. **Экспорт** — подготовка сценариев, карт и карточек к печати или демонстрации.
6. **Публикация и обсуждение** — дополнительный социальный слой для обмена материалами.

Социальные функции не должны подменять основную идею проекта. Друзья, личные сообщения и сообщества нужны для обсуждения и распространения материалов, но ядро системы должно оставаться полноценным даже без них.

## Что не входит в проект

Следующие возможности не являются обязательной частью ВКР и относятся к перспективному развитию:

- полноценный виртуальный игровой стол;
- видеосвязь и голосовые чаты;
- realtime-редактирование сценария несколькими пользователями одновременно;
- сложный боевой режим с автоматическим расчетом правил;
- поддержка конкретных коммерческих RPG-систем;
- хранение лицензированных материалов сторонних правообладателей;
- платные подписки и монетизация;
- мобильные приложения для iOS и Android;
- полноценная социальная сеть с лентой рекомендаций;
- сложная система рейтингов, достижений и репутации;
- автоматическая генерация сценариев искусственным интеллектом;
- продвинутая аналитика поведения пользователей.

Эти функции могут быть добавлены позже, но они не должны мешать завершению основного ядра проекта.

## Текущий статус

Проект находится в стадии активной разработки. Уже реализована основа клиент-серверного приложения, включая frontend, backend, базу данных, авторизацию, основные сущности и часть административного функционала.

### Реализовано базово

- Frontend-приложение на Next.js и React.
- Backend API на Laravel.
- Авторизация через Laravel Sanctum.
- Регистрация, вход, выход, профиль пользователя.
- Двухфакторная аутентификация.
- Кампании.
- Сценарии.
- Graph-first сценарный редактор.
- Graph-only backend/frontend сценариев на `scenario_nodes` и `scenario_transitions`.
- Карты.
- Персонажи.
- Предметы.
- Ассеты: файлы, изображения и токены.
- Универсальные связи между материалами через `entity_links`.
- Публикация материалов через `published_contents`: статусы `draft/published/archived`, видимость `private/unlisted/public` и public feed в разделе "Сообщество".
- PDF-экспорт сценария как graph-aware runbook по graph-модели.
- Жалобы на контент.
- Административные маршруты.
- Docker Compose окружение.

### Реализовано частично

- Графовый сценарный редактор: есть backend API, demo seed, typed contract, Typed Node Editors v1, Scenario Play/Preview Mode v2, Graph Canvas Navigation v2, Edge Editing v2, Edge Routing v1, Edge Ports v1, Selection UX v1, Undo/Redo v1, Readability v2, Node Presentation v2, Auto Layout v2, Minimap v1, Navigation Panel v2, inline labels, manual edge waypoints со сглаженными кривыми, computed obstacle-aware auto-routing, frontend-only graph validation v2 с errors/warnings и layout v2 с canvas как основной рабочей областью.
- Социальные разделы: сообщества, друзья, сообщения.
- Система приватности.
- Публикация материалов.
- Экспорт карт и карточек.
- Шаблоны для повторного использования.

### Требует переработки

- Graph UI должен быть развит дальше до publish/export enforcement и более полного graph layout/router.
- Backend legacy `chapters/blocks` удален; сценарии теперь graph-first на frontend и backend.
- Лут должен привязываться к предметам из реестра.
- Проверки уже имеют базовые исходы успеха/провала в graph contract; дальше нужны продуктовые правила невозможных переходов.
- Места должны связываться с локациями и картами.
- Сообщества должны стать системой публикаций, а не форумной заготовкой.
- Друзья и сообщения должны получить реальные backend-модели и API.
- Ассеты интегрированы в персонажей, предметы и карты через `entity_links.metadata.role`; сложные слои/resize токенов остаются следующим слоем.
- README, ТЗ и пояснительная записка должны отражать фактическое состояние проекта.

## Ключевые пользовательские сценарии

### 1. Подготовка кампании

Пользователь создает кампанию, указывает название, описание, жанр и дополнительные сведения. Внутри кампании он добавляет сценарии, карты, персонажей и предметы.

### 2. Создание сценария

Пользователь создает сценарий как граф сцен: добавляет узлы описания, диалога, боя, проверки, локации или лута, связывает их переходами и проверяет flow в режиме preview.

### 3. Связывание материалов

Пользователь связывает сценарий с персонажами, картами, предметами и другими материалами кампании. Это позволяет быстро находить нужные элементы во время подготовки или проведения игры.

### 4. Подготовка карты

Пользователь создает карту на сетке, задает размеры, размещает объекты и сохраняет структуру карты.

### 5. Подготовка карточек

Пользователь создает персонажей и предметы, которые в перспективе могут быть экспортированы в виде печатных карточек.

### 6. Экспорт сценария

Пользователь экспортирует сценарий в PDF для дальнейшего использования во время игровой сессии.

### 7. Публикация и обсуждение

Перспективный сценарий: пользователь публикует готовый сценарий, карту или шаблон в сообщество. Другие пользователи могут просматривать материал и оставлять комментарии.

### 8. Администрирование

Администратор управляет пользователями, жалобами, объявлениями и контентом, нарушающим правила платформы.

## Архитектура

Проект реализуется как **клиент-серверное веб-приложение** на основе модульного монолита.

### Общая схема

```text
Пользователь
    |
    v
Frontend: Next.js / React / TypeScript
    |
    v
Backend API: Laravel
    |
    v
PostgreSQL
```

Дополнительно используются:

- файловое хранилище для изображений и экспортированных файлов;
- очередь задач для тяжелых операций в перспективе;
- Mailpit для локальной проверки почтовых сообщений;
- Docker Compose для воспроизводимого запуска окружения.

### Почему выбран модульный монолит

Модульный монолит подходит для проекта ВКР, потому что он:

- проще в разработке и развертывании, чем микросервисы;
- позволяет разделить приложение на логические домены;
- сохраняет единую базу данных;
- подходит для одного разработчика;
- оставляет возможность дальнейшего расширения.

## Основные модули

### Auth & Profile

Отвечает за регистрацию, вход, выход, профиль пользователя, смену пароля, двухфакторную аутентификацию и получение текущего пользователя.

### Campaigns

Отвечает за создание и редактирование кампаний. Кампания является верхним контейнером материалов игрового мира.

### Scenarios

Отвечает за создание сценариев. Сценарии переведены в graph-first модель узлов и переходов поверх `scenario_nodes` и `scenario_transitions`; backend legacy-модель глав и блоков удалена.

### Scenario Builder

Основной модуль проекта. Сейчас поддерживает graph-first сценарный editor для связанных узлов: описание, диалог, место, проверка, лут, бой. Canvas занимает основную рабочую область, поддерживает pan, wheel zoom, fit-to-view, minimap-навигацию в верхнем UI overlay, ручной auto-layout v2 слева направо или сверху вниз с учетом размеров карточек, ветвлений и типов переходов, computed obstacle-aware auto-routing и separated input/output ports для читаемых directional edges от границ узлов, compact selected-only output handles, keyboard shortcuts `Escape`/`Delete`/`Backspace`, frontend-only undo/redo history для canvas-действий, semantic node type border styling, resizable node cards с content preview, drag-to-connect создание `linear` переходов, выбор transition, quick edit `type`/`label`, inline-редактирование label, manual edge waypoints со сглаженными кривыми и удаление transition. Режим `Превью` работает как read-only play mode: показывает сцену, typed-блоки узла, outcomes проверок, маршрут прохождения и связанные материалы. Список узлов, inspector и параметры сценария открываются как overlay/modal. Для graph-узлов есть typed editor форм по `config`, для переходов есть typed `condition`, frontend-only validation v2 с errors/warnings, а узлы можно связывать с картами, персонажами, предметами, ассетами, локациями, фракциями и событиями через `entity_links`. Legacy-редактор и backend слой глав/блоков удалены; следующий уровень развития — publish/export enforcement и более полный graph layout/router.

### Maps

Отвечает за создание карт на сетке, хранение размеров, клеток и объектов карты.

### Characters

Отвечает за создание персонажей, NPC и монстров.

### Items

Отвечает за создание предметов, их характеристик, редкости, стоимости и модификаторов.

### Assets & Tokens

Рабочий модуль библиотеки файлов: загрузка ассетов до 10 MB, типы `image`, `token`, `document`, `other`, preview для изображений/токенов, фильтрация, переименование, смена типа и привязка к кампании. Файлы хранятся через Laravel `public` disk. Ассеты можно назначать как портреты/токены персонажей, изображения предметов, фон карты и token palette карты.

### World

Рабочий модуль мира: локации, фракции и события доступны через отдельный API/UI, привязаны к владельцу и опционально к кампании. В интерфейсе есть вкладки, поиск, фильтр по кампании, создание, редактирование и удаление записей.

### Tags

Рабочий общий tagging-слой: пользовательские теги создаются один раз и привязываются к сценариям, картам, персонажам, предметам, ассетам, локациям, фракциям и событиям через `taggables`. В основных редакторах есть выбор, создание, переименование, удаление привязок и фильтрация по тегам. Legacy `campaigns.tags` остается отдельным полем кампаний.

### Relations

Рабочий слой универсальных связей: `entity_links` поддерживает направленные связи между сценариями, картами, персонажами, предметами, ассетами, локациями, фракциями и событиями. В редакторах есть блок "Связанные материалы", relation-типы `related`, `uses`, `located_in`, `member_of`, `rewards`, `mentions`, редактирование метки/типа связи, удаление и quick open целевого материала. Graph-node связи сохранены отдельным совместимым UX.

### Export

Отвечает за экспорт сценариев и игровых материалов.

### Community

Планируемый социальный модуль для публикации сценариев, карт, шаблонов и обсуждения материалов.

### Friends & Messages

Планируемый модуль для заявок в друзья, списка друзей и личных сообщений.

### Admin & Moderation

Отвечает за управление пользователями, жалобами, системными объявлениями и административными действиями.

## Модель предметной области

Базовая иерархия проекта:

```text
Пользователь
└── Кампания
    ├── Сценарии
    │   ├── Graph-узлы
    │   └── Переходы
    ├── Карты
    ├── Персонажи
    ├── Предметы
    ├── Локации
    ├── Фракции
    └── События
```

### Кампания

Кампания — верхний контейнер материалов. Она объединяет сценарии, карты, персонажей, предметы и другие сущности общей игровой вселенной.

### Сценарий

Сценарий — отдельное приключение, сессия или сюжетная ветка. Сценарий может быть связан с кампанией, но в перспективе должен иметь возможность существовать автономно.

### Узел сценария

Узел сценария — структурная единица сценария. В перспективной модели узлами могут быть:

- описание;
- диалог;
- место;
- проверка;
- лут;
- бой.

### Переход сценария

Переход связывает два узла сценария. Переход может быть линейным, условным, успешным, провальным или зависеть от выбора пользователя.

### Предмет

Предмет хранится в общем реестре и может быть связан с персонажем, сценой, лутом или событием.

### Карта

Карта хранит размеры, размер клетки, структуру объектов и ссылки на asset background/token objects в `data`. Canvas карты умеет показывать фон из image-ассета и размещать image/token ассеты как single-cell объекты.

Для запуска проекта требуется:

- Docker;
- Docker Compose;
- свободные порты `3000`, `8000`, `5432`, `8025`, `1025`.

```bash
docker compose down
```

## Безопасность

В проекте должны соблюдаться следующие требования безопасности:

- пароли хранятся только в виде хеша;
- доступ к приватным ресурсам проверяется на сервере;
- пользователь не должен получать доступ к чужим кампаниям, сценариям, картам и сущностям;
- административные маршруты доступны только администраторам;
- входные данные валидируются на backend;
- CORS настраивается только для разрешенных доменов;
- публичные текстовые данные должны экранироваться;
- загрузка файлов должна проверять тип, размер и права доступа;
- важные административные действия должны журналироваться;
- production-развертывание должно использовать HTTPS.

### Важное направление переработки авторизации

Текущая реализация использует access token на клиенте. В перспективе рекомендуется перейти к более чистой схеме:

- Laravel Sanctum SPA authentication;
- HttpOnly cookie;
- CSRF-защита;
- SameSite cookie;
- отказ от хранения access token в `localStorage`.

## Известные ограничения

- Часть социальных экранов пока является интерфейсной заготовкой.
- Сценарии теперь graph-first на frontend и backend: legacy-главы/блоки больше не редактируются в UI, а backend legacy-слой удален. Graph UI v2 поддерживает canvas-first раскладку, pan/zoom/fit-to-view, minimap в UI overlay, manual auto-layout v2 в двух направлениях, computed direct directional edges, separated input/output ports, keyboard clear/delete shortcuts, frontend-only undo/redo history, selected-only compact output handles, semantic node type border styling, resizable node cards, content preview, drag-to-connect созданием переходов, quick edit/inline label edit переходов, frontend-only validation v2 с errors/warnings и graph-aware PDF export.
- Graph Canvas содержит best-effort computed obstacle-aware auto-routing и auto-layout v2, но пока не содержит full obstacle router, persisted route points для auto-routes и полноценный graph layout engine; typed-модель условий/исходов и graph validation остаются frontend-only и не блокируют export/publish/backend-сохранение.
- Локации, фракции и события реализованы как самостоятельный World module v1, доступны для универсальных связей и могут привязываться к graph-узлам через inspector/preview/export.
- Библиотека токенов и изображений интегрирована в персонажей, предметы и карты через role-based asset links; resize/rotation/layers для map tokens еще впереди.
- Теги реализованы как пользовательский Tags module v1; campaign legacy tags не мигрированы в `taggables`.
- Экспорт карточек и карт требует доработки.
- Realtime-функции не являются обязательными для версии к защите.
- Проект ориентирован на подготовку материалов, а не на проведение полноценной онлайн-сессии.

## Итоговая формулировка границ проекта

**Кузница историй** — это не виртуальный игровой стол и не социальная сеть, а специализированная веб-платформа для подготовки сценариев настольных RPG. Основной результат проекта — рабочее пространство ведущего, где можно создать кампанию, подготовить сценарий, связать его с картами, персонажами и предметами, а затем экспортировать материалы для использования во время игры.

Сообщества, друзья и сообщения являются дополнительными модулями, которые помогают публиковать и обсуждать материалы, но не определяют основную ценность системы. Главный приоритет разработки — качественное ядро конструктора сценариев и связанных игровых сущностей.

## Graph visual metadata

Graph transitions separate gameplay rules from canvas visuals: `condition` stores typed gameplay conditions, while `scenario_transitions.metadata.visual.waypoints` stores manual edge waypoints rendered as smoothed canvas curves.
